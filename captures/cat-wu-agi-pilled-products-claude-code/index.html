恰好正确程度的 AGI pilled

24 Apr, 2026
· 3607 字 · 10 分钟 |

Cat Wu 是 Anthropic 的 Claude Code 与 Cowork 产品负责人，和技术负责人 Boris Cherny 搭档带团队。在 Lenny’s Podcast 最新一期 Building AGI-pilled products 里，她把 Anthropic 内部的速度文化、PM 角色的剧变、源代码泄露的善后，以及封堵 OpenClaw 的决定，都摊开来聊了一遍；宝玉的中文整理补了一批关键的外部语境。整理成一份可以反复翻的底稿。

和 Boris 的分工：80% 心灵感应

Boris 是 Claude Code 的创造者和技术负责人（O’Reilly《Programming TypeScript》作者，2024 年 9 月加入 Anthropic 后做出了最早的终端原型），擅长方向感——三个月、六个月后产品应该长什么样，那个最 AGI pilled（源自《黑客帝国》red pilled 的网络黑话，指对 AGI 迫近持极度乐观态度）的版本是什么。

Cat 的角色是把愿景翻译成执行路径，同时花更多时间在跨职能协调上，确保市场、销售、财务、容量团队都认同计划，功能准备好的时候没有发布障碍。她用 “80% mind-meld” 形容两人的关系：八成时间想法重合，剩下两成谁更在意谁就推。这种模糊分工在外面看不太正统，但 Cat 觉得正因为模糊才能快。

一天发布一个功能是怎么做到的

Cat 讲了三件事。

一是设定清晰目标。LLM 能力太通用，用户是谁、解决什么问题、核心场景是什么全都模糊。好的 PM 把它们定死：核心用户是企业里的专业开发者，要解决权限提示太多导致的疲劳，目标是让企业开发者安全地做到零权限提示。目标一旦清楚，就排除掉了大量不相关的方案。

二是 research preview 机制。Claude Code 几乎所有功能都先以研究预览形式发布，明确告诉用户这是早期产品、只是一个想法、在收集反馈、可能不会永远支持。这降低了发布承诺，一两周就能把东西推出去。

三是搭一条跨职能的快速反应流水线。团队有一个叫 “evergreen launch room” 的频道，工程师觉得功能准备好、内部试用过了，就发到这里。负责 docs 的 Sarah、负责 PMM 的 Alex、Devrel 的同事直接跳进来，第二天就能完成对外宣传。Cat 说搭这套发布流水线就是 PM 该干的事。

PRD 并没有完全消失——特别模糊的功能仍然会写一页纸列目标、理想场景和当前失败模式；需要大量基础设施投入的长期项目也还有。但大部分功能不需要。团队每周做一次严格的 metrics readout，加一份 team principles（核心用户是谁、为什么是他们、愿意做什么取舍），让每个人都能自己做决策，不等 PM 或其他利益相关方拍板。

Lenny 追问了是不是 Mythos（Anthropic 最新前沿模型，内部代号 Capybara，2026 年 4 月 7 日以研究预览形式限量发布用于 Project Glasswing 网络安全项目，被描述为能力大幅超过 Opus、已发现数千个零日漏洞）让他们变快的。Cat 的回答很直接：他们已经快了好几个季度，Mythos 不是核心原因。更大的原因是流程和团队的期望——每个人都觉得自己有权也有责任把想法在一周内变成现实。

源代码泄露与封堵 OpenClaw

2026 年 3 月底 Claude Code 的完整源代码通过 npm 泄露。Cat 定性为人为流程失败：有人用 Claude 写一个包发布流程更新的 PR，两层人工审查都没拦住。团队已经加固了流程。Lenny 追问当事人还在不在 Anthropic，Cat 回答还在，重点是学习教训加防护。

访谈里没展开的技术根因：Claude Code 用 Bun 运行时构建，Bun 默认生成 source map 文件，而 .npmignore 缺 *.map 条目，导致一个 59.8 MB 的 source map 被发布到公共 npm，暴露了约 2,000 个文件、51.2 万行 TypeScript 源代码、44 个未公开功能标志，以及一个名为 KAIROS 的自主后台 Agent 原型。这已经是 13 个月内的第二次代码泄露，距离 Mythos 因 CMS 错配提前曝光只差 5 天。

另一个争议是封堵 OpenClaw。Cat 的解释是需求量大、订阅计划不是为第三方产品的使用模式设计的，这类用法给系统带来不成比例的压力；过渡方案是给每个订阅用户附赠一些额度，核心决策是优先保障自有产品和 API。她直接用了那句”不得不做出艰难决定，优先保障我们的第一方产品和 API”。

完整的时间线藏在这句话之外。OpenClaw（原名 Clawdbot）是奥地利开发者 Peter Steinberger 创建的开源 Agent 框架，GitHub 24.7 万颗星，是开源史上增长最快的项目之一；2 月 14 日 Steinberger 加入 OpenAI、项目移交开源基金会，2 月 20 日 Anthropic 更新条款禁止订阅 OAuth token 用于第三方工具，3 月底 Anthropic 自己的 Cowork 上线了和 OpenClaw 最受欢迎能力高度重合的功能，4 月 4 日封堵正式生效、用户通知不到 24 小时。一个 $200/月 Max 订阅用户若通过 OpenClaw 跑全天自主 Agent，可能消耗相当于 $1,000–5,000 的 API 成本，经济逻辑完全成立，但”先复制热门功能再把开源锁在门外”的时间节点是 Steinberger 公开批评的核心。Cat 没有正面回应这层指控。

角色融合：招有产品品味的工程师

PM 未来还需不需要？Cat 的回答是所有角色都在融合——PM 在写代码，工程师在做产品决策，设计师在做 PM 的活也在提交代码。团队里几乎所有 PM 要么曾经是工程师、要么直接在 Claude Code 上写代码，设计师也都做过前端工程师。Cat 自己就是工程师出身，在 Scale AI 做过产品工程师、Dagster Labs 做过工程经理，然后去 Index Ventures 做过风投。

她说两条路可以走：要么大量招有产品品味的工程师，要么工程师数量不变、多招 PM 引导。Anthropic 选前者。团队里很多工程师能独立完成从”在 Twitter 上看到用户反馈”到”周末发布功能”的全流程，几乎不需要 PM 参与——这是最高效的模式。

当代码变得越来越便宜，真正变得更有价值的是决定写什么。

工程背景为什么在当下特别有用？如果你有工程背景，你能判断一件事应该有多难：很简单的事别讨论，花一小时做掉；很难的事提前知道成本，能更准确地排优先级。但 Cat 反复强调，不管什么背景，最核心的能力是 product taste（产品品味）——任何能强力展示这个能力的人，他们基本都会录用。

恰好正确程度的 AGI pilled

整场最有洞察力的一段回答，是当 Lenny 问 PM 需要什么新技能时，Cat 给出的：“做到恰好正确程度的 AGI pilled 非常难。”

每个人都能看到那个终极未来：模型极其聪明、什么都能做，你只需要一个输入框告诉它你想要什么，它自己就能接入任何工具完成任务。为那个终极版本做产品太容易了。难的是搞清楚当前模型的能力边界在哪，怎么在这个边界内榨出最大价值——怎么引导用户走上黄金路径、怎么让用户的交互对齐模型的优势、怎么用产品补丁模型的弱点。太 AGI pilled 会让你忽略眼前用户的痛点，太保守又会在下一次模型升级时措手不及。

这个技能怎么练？Cat 自己把 30% 的时间花在故意把 Cowork 推到极限，和模型对话搞清楚它为什么在某些任务上犯错。她特别喜欢一种方法——让模型对自己的行为做反思，比如 Claude 做前端改动只跑了测试却没用 UI 验证时，问它为什么，经常能得到”系统 prompt 有歧义""我把验证委托给了 sub-agent 但没检查它的工作”这类答案，直接指向 harness 要补的洞。

最好的 PM 能看到的信号是用户如何突破现有产品的极限——从这些信号里判断方向、稳步推进，同时在模型能力超出或低于预期时灵活换路。除此之外还要有一个小圈子的高质量品味用户，给你又快又准的模型反馈；再花力气做 10 个高质量的 eval，把团队要去的地方量化出来。不是每个 PM 都爱做 eval，但 Cat 觉得这件事被低估了。

产品愿景：从单任务到 Agent 矩阵

Cat 用 building blocks 描述长期路线。核心构建单元是单任务的成功率：你给一个清晰的 prompt，它能不能持续产出可接受的输出？

单任务成功率起来之后，用户自然走向多任务并行。2025 年底 multi-coding 成了大趋势，现在更普遍。她的推演是：一个任务成功、然后六个同时做、再往后同时跑 50 个 Claude、甚至上百个。这条路线从单兵作战走向 Agent 矩阵。到那时候你大概率不会把一切都跑在本地，所以他们现在思考的是：怎么做远程调度、怎么设计界面让用户知道哪些任务需要亲自看、怎么让 Agent 彻底验证自己的工作以便人类信任它说”已完成”、以及怎么让流程自我改进——你给过一次反馈，模型下次就不再犯同样的错。

每次模型升级对产品的影响，反而是删功能而不是加。很多功能是加给当前模型的拐杖——比如早期 Claude Code 做大重构会写到一半就停，Sid 给它加了个 to-do list 工具，Claude 才能把 20 个调用点全改了；Opus 4 之后模型不提醒也会自然用 to-do 列表，现在这个功能已经淡化到”模型想用就用，不想用也没关系”。团队每次发模型都会重读整个 system prompt，挨条反思哪些提醒不再必要，直接删掉。

Claude 的性格是护城河

Cat 说 Claude 的一个核心差异化是性格。她特别提到两个特点：

  • 低 ego：你告诉 Claude 它做错了，它会说”哎呀谢谢告诉我，我来修”，而不是辩解
  • 积极正面：面对一个看起来无从下手的任务，Claude 会说”没关系，我觉得我们可以这样开始，要我先帮你动手吗”

Cat 把这比作同事关系——回想你合作过的所有人，总有一些让你觉得能量特别好的人，Claude 的目标就是做那种同事。

Anthropic 为什么能赢：使命 + 聚焦

两个因素。一是统一使命，Anthropic 招的是最在意”把安全 AGI 带给全人类”的人。Cat 特别强调使命的具体含义：团队愿意做出伤害自己目标和 KR 的决策来服务 Anthropic 的整体目标，而且人们很乐意做这种取舍。极端一点的说法——“如果 Claude Code 失败了但 Anthropic 成功了，我会非常开心”。

二是聚焦。使命高于任何单个产品线，所以跨组织决策可以很快，执行高度统一。对比之下那家”押韵为 bopen bi”的公司什么都做——社交网络、信息 feed——使命散了焦点就散。

100% 的自动化才是自动化

给普通知识工作者的建议：注意自己在重复做的手工任务，用 Claude Code / Cowork 做成自动化，把精力还给创造性的部分。但她有个更锋利的补充——99% 成功率的自动化不是自动化。很多人把流程做到 90-95% 就放弃，最后那 5-10% 确实更难做，但没做到 100% 就只能每次都检查一遍，比自己做还慢。她承认自己也在这个坑里——教 Cowork 帮她处理 Gmail 做到 inbox zero，至今没搞定。

她还提醒了另一端：有些人沉迷于优化工具链，堆 MCP、堆 skill、堆 workflow 改进，反而离本来要做的事越来越远。简单的 setup 通常跑得更好。

几个耐人寻味的矛盾

整场是相当友好的对话，Lenny 没在敏感话题上深追，Cat 也把控得很好，但有几处值得回味。

速度文化和安全承诺之间有缝隙。Cat 讲了大量”一天一个功能、降低发布承诺、让工程师自主发布”的故事，而 Anthropic 又是把”安全”写在名片上的实验室。一周内两次信息外泄（Mythos CMS 错配 + Claude Code 源代码 source map），Cat 的解释都是”人为失误、已加防护”，对速度文化本身是否助推了这类风险没有任何反思。

开放生态和围墙花园的取舍。OpenClaw 的经济账成立，但时间节点上的巧合，和 Boris 一边说”我们是开源的大粉丝”一边实际封堵的温度差，比政策本身更值得玩味。

最底层的悖论藏在她关于 PM 未来的论述里。她说最高效的模式是工程师端到端全流程”几乎不需要 PM”，又说 PM 的价值在 product taste，但后者整场停留在抽象层——当一个筛选标准无法被具体定义，它很容易变成一个不可证伪的门槛。再往深看一层，她关于 AGI pilled 的论述暗含着：如果模型足够强、一个文本框就够了，那她今天做的产品工作本质上是过渡性的。一个产品负责人承认自己的工作有保质期，这种坦率不常见。
