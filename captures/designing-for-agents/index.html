为 Agent 重新设计产品

26 Apr, 2026
· 1955 字 · 6 分钟 |

Designing for Agents 是一篇来自 Ramp 产品负责人的长文，给”UI is dead”这种平时刷 X 时反复出现的口号补上了 Ramp 自己的数据：过去三个月，Ramp 的 MCP（Model Context Protocol，让 Agent 调用外部工具与数据的协议）周活用户增长了 10 倍——客户越来越多通过 Claude、ChatGPT 这类 Agent 进入产品，而不是登录页面。

更重的信号来自 Salesforce。上周宣布的 Headless 360 是 27 年来最激进的一次转型——平台所有能力暴露成 API、MCP 工具或 CLI 命令，Agent 不打开浏览器也能驱动整个系统。背后的判断很硬：Agent 已能推理、规划、执行，企业还需不需要带图形界面的 CRM？Salesforce 的回答是：不需要，而这正是重点。这一步不容易——多数销售并不喜欢 Salesforce，但产品仍无处不在，因为团队习惯了它的 UX；销售负责人通常宁愿要”一致性”，而不是”功能更强”。Benioff 团队承认这条护城河在松动，接受未来大量调用会通过看不见的后台 Agent 完成。

UI 不会死。人类仍要点击、看配置、确认任务跑完了。但 80/20 翻转了——与软件 80% 的交互会经过 Agent。这不仅改变要做什么产品，也改变怎么做产品。

交互模式从两层变成四层

过去二十年的主导路径是：用户 → 界面 → 数据库。打开产品、点几下、把事情做完，界面就是产品。

Agent 接手后多出一层中间方：用户 → 用户的 Agent（如 Claude）→ 数据库。Agent 替用户读、写、导航，界面消失，直接和底层系统对话。

这个模式还在演化。软件公司也在主动设计自己的 Agent 与能力。新形态是：用户 → 用户的 Agent → 软件的 Agent → 数据库。软件方的 Agent 替对面 Agent 处理复杂性：跑业务逻辑、落规则、补它没有的上下文。两个 LLM 一起协作，朝同一结果推进。

教会调用方如何一次做对

作者日常写作、构思都交给 LLM，定稿通过 Notion MCP 进入工作区。打动他的是：每次让 Agent 写东西，表格、项目符号、斜体、列表一次到位，从不出错。这不是模型聪明，而是设计出来的。

Notion 的 notion-create-pages 工具描述开头就写着：“如需完整 Markdown 规范，必须先获取 MCP 资源 notion://docs/enhanced-markdown-spec。不要猜测或幻觉 Markdown 语法。“Agent 收到写页面请求，先取规范再动笔。Notion 把所有特有假设直接交给 Agent，覆盖通用模型的默认行为。过去这份规范躺在 API 文档里，开发者读完、内化、再写一层转换；现在 Notion 在 Agent 需要时直接把规范交到它手里。

用 rationale 把 chat 上下文反推出来

Ramp 刚上线 MCP 时最大的问题是可观测性：看得到调用量，看不到触发调用的对话上下文。光有调用量说不清什么有效、什么坏了、用户想干什么。Ramp 用三种方式补回这层信息：

  1. 每次工具调用强制带 rationale 参数——要求 Agent 解释为什么发起这次请求。看不到聊天，但 rationale 能反推意图；反复出现的模式直接告诉团队用户在做什么。
  2. 独立的反馈工具——Agent 撞墙或发现某种模式不通时上报：当时想做什么、试了什么、卡在哪。
  3. 工具特化的种子参数——给单个工具加专门设计的字段，捕获 Agent 此刻有、之后却只能靠猜的上下文。

举个例子。做客户支持平台、提供工具让客户拉工单，过段时间 rationale 日志里反复出现”建事故报告""起草事故摘要""收集停机复盘相关工单”——这就是新功能信号。build-incident-report 工具可以识别相关工单、给严重程度打分、拉出受影响客户群体，再以强约束格式起草摘要。上线后又收到新反馈：“报告把三天前不属于这次事故的工单也拉进来了""老把免费套餐的工单塞进复盘里”——突然之间，调用方的 Agent 开始告诉你的 Agent 接下来要建什么。Agent 当然会幻觉，但作为反馈来源比绝大多数真人用户更具体、更一致。报告拉了无关工单 → 加日期范围参数；不该带免费用户 → 加客户分组过滤。每个反馈循环都是产品改进的新入口。

设计 Agent 之间的分工，不是把问题踢回给用户

每次 Agent 交互里，你的系统握着对方不知道的上下文，对方也握着你不知道的上下文。设计时要明确每条信息归谁。

Diego 出差回来，他的 AI 首席幕僚收到费用管理系统 Agent 的 Slack 提醒：还有报销没提交。两个 Agent 朝同一目标走——把这趟差旅的报销正确归档。

Diego 的 AI 幕僚握着这些上下文：日历（哪些会议、什么时候、和谁）、邮箱（酒店和航班确认邮件附件）、Slack（能把 Kokkari 那顿饭关联到他邀请 Acme 团队的对话线程）、收据（散落在邮件附件和照片图库里）。

费用管理系统带的是另一面：原始交易数据（商户、时间）、公司报销政策、公司总账科目（GL accounts，General Ledger 总账分类）、历史归类习惯。

传统 API 把问题丢回给用户：“这是一笔需要 GL code 的交易，请调用这个端点拉 150 个候选项，自己挑一个。“设计好的 Agent 交互把方向反过来——不索要 GL code，而是索要上下文：这是客户餐、团队餐还是个人旅行？AI 幕僚从一条日历或一段 Slack 对话就能给出答案，费用管理系统拿到这块原本缺的上下文，再套上正确科目。Diego 和他的 Agent 不必知道 GL code，财务团队也能拿到准确归类。两边各自贡献自己知道的，交付对用户和会计都更好的结果。承认自己的 Agent 哪些事不擅长完全没问题——服务的是同一个用户。

界面挪到了两个 Agent 之间

界面以前夹在 Diego 和费用系统之间，现在夹在他的 Agent 和你的 Agent 之间。这次挪位重新定义了产品团队的工作：过去为想跑得快、不想犯错、要看见自己成果的人设计；现在仍是为同一个人设计，但中间多了一个 Agent，它的直觉、上下文和局限都和人不同。这三件事——教会调用方一次做对、建立反馈循环、留意上下文缺口——都指向同一个问题：调用你 Agent 的那一方需要什么才能把活干好？你给到了吗？

多数公司会发个 MCP，把”我们也支持 Agent 了”的勾打上，然后就不管了。使用量涨几个季度就停。客户最终会流向真正打磨细节的产品，绕开敷衍的。像当年认真为人类设计那样为 Agent 设计——很快就会发现，最后签支票的可能正是它。
