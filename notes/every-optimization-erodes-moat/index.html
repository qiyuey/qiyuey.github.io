每次优化都在拆掉自己的护城河

Updated: 29 Apr, 2026
· 1693 字 · 5 分钟 |

自做项目的每一次方向调整，都应该问一句：这次调整让我能掌握的空间更大了，还是更小了？

2026 年初我想给 3 岁女儿做一个语音陪伴硬件，让她在不接触屏幕的前提下接触 AI。从立项到放弃，前后经历了四次转向。每一次都觉得自己在”想得更清楚”，回头看其实是在持续拆掉自己的护城河，直到没有任何东西值得继续做。

四次转向

第一版：自己拼 ASR + LLM + TTS

最初的方案是经典三段式管线：树莓派级别的 SBC、麦克风采集、云端 ASR 转文字、调 LLM API、TTS 合成、喇叭播放。安全策略由我自己设计——前置过滤、后置过滤、儿童模式 Prompt、家长开关。

这个版本最重要的是：每一段都是我能掌握的。链路怎么编排、上下文怎么裁剪、什么时候降级、敏感意图怎么拒答——都是项目本身的判断空间。技术栈复杂，但复杂度本身就是壁垒。

第二版：换成多模态 S2S 模型

接着我意识到三段式串联的延迟通常落在 1.5-3 秒，3 岁孩子等不了；TTS 合成的语调也很难做出真正温柔的”陪伴感”。S2S 模型可以做到亚秒级首字响应、保留语气情绪。

换模型的决定本身没问题。但代价是：原本”管线编排”这个我能做深的环节，让渡给了模型本身。 我从”设计三段式怎么协作”变成了”调用一个 API”。安全策略也被迫退化——失去了文本中间产物作为审查锚点，只能加一路影子转写做后审。这是从”我设计护栏”退到”我监督模型”。

第三版：放弃专用硬件，复用旧手机

再下一跳是发现专门做无屏硬件利用率太低——一台只能听不能看的设备，能力天花板被自己锁死了。复用旧手机看起来全是好处：麦克风、喇叭、摄像头、屏幕、电池、Wi-Fi 全部现成，还可以加图片识别和视频，把项目从”语音玩具”升级成”多模态启蒙伙伴”。

但这一跳同时也意味着：硬件方案这个差异化也被让渡了。 我从”设计一个能持续陪伴的物理形态”变成了”在 Android 上写一个 Kiosk App”。原本”为什么是这种形状、这种交互”的产品判断，被”用现成手机就行”消解掉了。

第四版：放弃

到这里我突然停住了——剩下要做的事就是”在 Android 上调一个 S2S API、加一层 Kiosk、配一份儿童 Prompt”。每一项小天才手表都已经做了。我能加的差异化，只剩下”它是我自己写的”这个心理价值。

不是不能继续，是继续下去没有真正属于我的那部分。

演化路径

阶段我掌握的核心让渡给了还剩什么
ASR + LLM + TTS三段管线编排、安全前后置过滤、Prompt无全部
S2SPrompt、影子审查、双工连接管线编排、TTS 音色、安全锚点大半
复用手机App 体验、Kiosk 配置、Prompt硬件形态、电源/续航、音频质量一小半
放弃前Prompt、家长配置几乎全部与现成产品的差距趋近于零

每一行都是合理决策。问题在合并起来看：每次都在用”做得更好”的逻辑去换掉自己原本能掌握的部分。当所有”做得更好”的选项都指向”用现成的”，项目本身就走到了无价值区。

模式提取

回头看，每次决策时本应同时问两个问题：

  1. 这样改，做得更好吗？——我每次都在认真回答这个
  2. 这样改，我自己掌握的空间是变大了还是变小了？——这个问题从来没问过

商业语境里有个贴近的框架——Geoffrey Moore 的 core 和 context：core 是创造差异化、让别人选择你的部分，context 是必要但不差异化的部分，应该外包给现成方案。Clayton Christensen 的”价值守恒定律”描述同一过程的另一面：商品化让一层失去利润，价值迁移到相邻的稀缺层。

但个人项目的 core 和商业项目的 core 不是同一件事。 商业项目里的 core/context 判断，是为了把资源留给市场差异化；个人项目里的 core/context 判断，是为了把精力留给自己能增长判断力的地方。对个人项目来说，被优化掉的如果正是这部分，项目即使更可交付，也已经失去了继续做的理由。

“做得更好”的逻辑之所以会一路滑向”全用现成的最好”，是因为它默认了”更好”等于市场标准下的更好。但个人项目的标准本来就不在市场上。

适用边界

  • 这条判断只针对自做项目。商业项目逻辑相反——尽量用现成的，把精力留给真正的差异化
  • “做一遍”本身有学习价值。如果目的是练手，即使最后没用也不算亏。这条 note 不否定那种动机，只是提醒：练手项目要承认自己是练手项目，不要假装是产品
  • 不是说不能用现成模块。问题是当所有关键模块都被现成的吃掉、自己只剩”调用者”角色时，要敢于承认项目已经空心化
  • 这条逻辑在公司层面也成立。2023-2026 死掉的几千家 AI wrapper 创业公司是同一个规律的商业版——区别只在替代者不是小天才手表，是模型供应商的下一个 release

判断框架的更新

下次启动一个自做项目时，每个方向调整点都要回头看一眼”掌握空间地图”。如果连续两次调整都在让地图缩水，应该停下来重新审视，而不是继续优化下去。

一个可以日常自问的犀利问题：拿掉那些”现成的”，剩下的还是不是我的？ 拿掉模型、拿掉 API、拿掉平台、拿掉硬件——还剩什么是别人没法在一个周末做出来的？这个问题对 AI wrapper 公司是死亡测试，对个人项目同样适用。

放弃不是失败。把判断做透到能干脆放弃的程度，比硬撑着做完一个空心项目，更接近”想清楚了”这件事本身的价值。
